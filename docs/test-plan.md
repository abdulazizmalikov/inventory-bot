# План тестирования

Этот документ описывает шаги для тестирования Telegram бота управления складом.

## Предварительные требования

1. ✅ Бот создан и токен получен
2. ✅ Google Sheets настроен с правильной структурой
3. ✅ Service Account создан и имеет доступ к листу
4. ✅ .env файл заполнен корректно
5. ✅ Бот запущен (`npm run dev`)

## Тест 1: Проверка доступа

**Цель:** Убедиться, что только администратор может использовать бота.

**Шаги:**
1. Откройте Telegram и найдите вашего бота
2. Отправьте `/start`
3. **Ожидаемый результат:** Бот отвечает главным меню с кнопками
4. Попросите другого пользователя (не в списке ADMIN_IDS) отправить `/start`
5. **Ожидаемый результат:** Бот отвечает "У вас нет доступа"

**Статус:** ⬜ Не выполнено / ✅ Выполнено

---

## Тест 2: Создание заказа (Flow A)

**Цель:** Проверить создание нового заказа у поставщика.

**Шаги:**
1. Нажмите "🚚 В пути" → "➕ Новый заказ"
2. Выберите поставщика из списка (или создайте нового)
3. Выберите SKU из списка
4. Введите количество (например, 10)
5. Нажмите "✅ Продолжить"
6. Введите сумму заказа (например, 1000000)
7. Введите стоимость доставки (например, 50000)
8. Проверьте подтверждение и нажмите "✅ Подтвердить"

**Ожидаемый результат:**
- Заказ создан с ID
- В Google Sheets:
  - В `purchase_orders` добавлена строка со статусом `IN_TRANSIT`
  - В `purchase_order_items` добавлены товары
  - В `movements` добавлены записи типа `PO_CREATE`

**Проверка в Sheets:**
```sql
-- Проверьте последний заказ
SELECT * FROM purchase_orders ORDER BY created_at DESC LIMIT 1;
```

**Статус:** ⬜ Не выполнено / ✅ Выполнено

---

## Тест 3: Получение заказа (Flow B)

**Цель:** Проверить получение заказа и перемещение в офис.

**Предварительно:** Создайте тестовый заказ (Тест 2)

**Шаги:**
1. Нажмите "🚚 В пути" → "✅ Получить заказ"
2. Выберите заказ из списка
3. Проверьте детали заказа
4. Нажмите "✅ Получить"

**Ожидаемый результат:**
- Заказ помечен как `RECEIVED`
- В `movements` добавлены записи типа `PO_RECEIVE`
- В `office_stock` обновлены остатки (увеличены)

**Проверка:**
- Откройте "🏬 Склады" → "🏢 Офис"
- Должны отображаться полученные товары

**Статус:** ⬜ Не выполнено / ✅ Выполнено

---

## Тест 4: Отправка товара (Flow C)

**Цель:** Проверить отправку товара на маркетплейс.

**Предварительно:** Убедитесь, что в офисе есть товары (Тест 3)

**Шаги:**
1. Нажмите "📤 Отправка"
2. Выберите маркетплейс (Uzum или Yandex)
3. Выберите SKU
4. Введите количество (меньше или равно остатку в офисе)
5. Подтвердите отправку

**Ожидаемый результат:**
- В `movements` добавлена запись типа `SHIP`
- В `office_stock` уменьшены остатки
- Маркетплейс НЕ обновляется (только внутренний учет)

**Проверка:**
- Откройте "🏬 Склады" → "🏢 Офис"
- Остатки должны уменьшиться

**Статус:** ⬜ Не выполнено / ✅ Выполнено

---

**Проверка:**
- Откройте "🏬 Склады" → "🏢 Офис"
- Остатки должны увеличиться

**Статус:** ⬜ Не выполнено / ✅ Выполнено

## Тест 6: Просмотр складов (Flow E)

**Цель:** Проверить отображение остатков на всех складах.

**Шаги:**
1. Нажмите "🏬 Склады"
2. Проверьте каждый раздел:
   - "🚚 В пути" - должен показывать заказы в статусе IN_TRANSIT
   - "🏢 Офис" - должен показывать остатки в офисе
   - "🛒 Маркетплейс" - должен пытаться загрузить данные из API
   - "📊 Итого" - должен показывать сумму всех складов

**Ожидаемый результат:**
- Все разделы отображают корректные данные
- Если API не настроен, маркетплейс показывает соответствующее сообщение

**Статус:** ⬜ Не выполнено / ✅ Выполнено

---

## Тест 7: Обновление продаж и расчет прибыли (Flow F)

**Цель:** Проверить импорт продаж и расчет прибыли.

**Предварительно:** Настройте Uzum API (или используйте заглушку)

**Шаги:**
1. Нажмите "🔄 Обновить продажи"
2. Выберите период (например, "Последние 7 дней")
3. Дождитесь обработки

**Ожидаемый результат:**
- Продажи импортированы в `uzum_sales_raw`
- Рассчитана прибыль в `profit_calc`
- Отображается сводка:
  - Общая выручка
  - Общая себестоимость
  - Общая прибыль
  - Топ-10 товаров по прибыли

**Проверка в Sheets:**
- Проверьте лист `uzum_sales_raw` - должны быть новые записи
- Проверьте лист `profit_calc` - должны быть расчеты

**Статус:** ⬜ Не выполнено / ✅ Выполнено

## Тест 8: Частичная отправка

**Цель:** Проверить, что можно отправить часть товара.

**Предварительно:** В офисе должно быть минимум 10 единиц товара

**Шаги:**
1. Отправьте 5 единиц товара (Тест 4)
2. Проверьте остаток в офисе
3. Отправьте еще 3 единицы того же товара

**Ожидаемый результат:**
- Обе отправки успешны
- Остаток в офисе уменьшается на каждую отправку
- В `movements` две записи типа `SHIP`

**Статус:** ⬜ Не выполнено / ✅ Выполнено

## Дополнительные проверки

### Проверка целостности данных

1. Проверьте, что сумма в `purchase_orders.total_amount_uzs` = `order_amount_uzs + shipping_cost_uzs`
2. Проверьте, что `office_stock` соответствует сумме движений типа `PO_RECEIVE` минус `SHIP` плюс `RETURN`
3. Проверьте, что все `movements` имеют корректные типы и направления

### Проверка производительности

1. Создайте заказ с 20+ товарами
2. Проверьте время отклика бота
3. Проверьте время загрузки списка складов при большом количестве SKU

## Известные ограничения

1. **Uzum API:** Если API не настроен, маркетплейс будет показывать 0 или ошибку
2. **Пакетная обработка:** Большие списки SKU могут требовать пагинации (уже реализовано)
3. **Одновременные операции:** Бот не блокирует одновременные операции от одного пользователя (для MVP это приемлемо)
